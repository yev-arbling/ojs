# OJS v1.0 Field Coverage Backport — Design Spec

**Date**: 2026-05-24
**Branch**: `feat/v0.1-field-coverage-backport`
**Status**: Draft

## Motivation

OJS v1.0 was designed top-down from standards (GIA, CIBJO, ISO, GMC, ACP). OJS v0.1 was built bottom-up from 185,000 real products across 86 jewelry brands via crawler pipelines. Comparing them reveals that v1.0 dropped ~17 practically useful fields and ~18 enum values that real stores use daily.

This spec backports those gaps into v1.0 without compromising its modular architecture.

## Scope

Three workstreams:

1. **Enum additions** — 20 new values across existing and new enums
2. **Field additions** — 16 new fields across 5 modules
3. **Structural improvement** — extract all inline enums to separate codelist files

## 2. Field Additions

### 2.1 IdentityModule — 2 fields

#### `product_subtype` (enum)

```python
class ProductSubtype(str, Enum):
    # Rings
    ENGAGEMENT_RING = "engagement_ring"
    WEDDING_BAND = "wedding_band"
    FASHION_RING = "fashion_ring"
    SIGNET_RING = "signet_ring"
    ETERNITY_RING = "eternity_ring"
    COCKTAIL_RING = "cocktail_ring"
    PROMISE_RING = "promise_ring"
    MENS_RING = "mens_ring"
    # Earrings
    STUD = "stud"
    HOOP = "hoop"
    HUGGIE = "huggie"
    DROP = "drop"
    EAR_CUFF = "ear_cuff"
    CLIMBER = "climber"
    # Necklaces
    CHAIN = "chain"
    CHOKER = "choker"
    PENDANT_NECKLACE = "pendant_necklace"
    COLLAR = "collar"
    LAYERED = "layered"
    # Bracelets
    BANGLE = "bangle"
    CUFF = "cuff"
    TENNIS = "tennis"
    CHARM_BRACELET = "charm_bracelet"
    CHAIN_BRACELET = "chain_bracelet"
    # General
    OTHER = "other"
```

**Location**: `IdentityModule.product_subtype: Optional[ProductSubtype]`
**Rationale**: Every marketplace requires product_subtype for faceted navigation. product_type alone cannot distinguish engagement ring from signet ring.

#### `note` (string)

```python
note: Optional[Annotated[str, Field(max_length=1000)]] = Field(
    default=None,

    "E.g. 'All natural stones vary slightly in color', 'Runs small, size up'."
)
```

**Location**: `IdentityModule.note`
**Rationale**: Distinct from description (marketing copy). Legal disclaimers, sizing quirks, material notes.

### 2.3 StonesModule.Stone — 2 fields

#### `count` (integer)

```python
count: int = Field(
    default=1,
    ge=1,
    description="Number of stones of this type in the piece. "
    "Use for pavé/accent stones: {species: diamond, count: 24, total_carat: 0.48}."
)
```

#### `total_carat` (number)

```python
total_carat: Optional[Annotated[float, Field(gt=0)]] = Field(
    default=None,
    description="Total carat weight for all stones of this type when count > 1."
)
```

**Rationale**: Without count, a pavé ring with 24 diamonds needs 24 identical Stone objects. Every retailer describes stones as "24 round brilliant diamonds, 0.48 tcw".

**Interaction with existing fields**: `StonesModule.total_carat_weight` remains as the overall sum across ALL stone entries; `Stone.total_carat` is per-entry (for that group of identical stones).

## 3. Structural Improvement: Enum Extraction

### Problem

v1.0 has ~40 enums. Only 8 have separate codelist files (`spec/v1/codelists/`). The rest are inline in Python models and compiled into the 6,088-line `ojs-strict.json` monolith.

### Solution

Extend `src/python/ojs/generate_codelists.py` to export ALL enums to individual codelist JSON files:

```
spec/v1/codelists/
├── eras.json              (existing)
├── gems.json              (existing — split into species, cuts, clarity, treatments)
├── hallmarks.json         (existing)
├── metals.json            (existing — types, colors, finishes, platings)
├── pearls.json            (existing)
├── settings.json          (existing)
├── styles.json            (existing)
├── treatments.json        (existing)
├── audiences.json         (NEW)
├── availability.json      (NEW)
├── biocompatibility.json  (NEW)
├── chain-styles.json      (NEW)
├── cleaning-methods.json  (NEW)
├── closures.json          (NEW)
├── colored-stone-clarity.json (NEW)
├── conditions.json        (NEW)
├── confidence-sources.json (NEW)
├── connectivity.json      (NEW)
├── earring-post-types.json (NEW)
├── edition-types.json     (NEW)
├── grading-labs.json      (NEW — extracted from gems.json or separate)
├── image-roles.json       (NEW)
├── motifs.json            (NEW)
├── occasions.json         (NEW)
├── placement-anchors.json (NEW)
├── price-tiers.json       (NEW)
├── product-types.json     (NEW)
├── regulatory-flags.json  (NEW)
├── source-systems.json    (NEW)
├── product-subtypes.json     (NEW)
└── threading-types.json   (NEW)
```

### Format

Each codelist file follows the existing v1.0 pattern:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://schema.openjewelryschema.org/v1/codelist/<name>.json",
  "title": "OJS codelist: <name>",
  "description": "<description>",
  "version": "1.0.0",
  "values": [
    { "code": "value_1", "label": "Human Label" },
    { "code": "value_2", "label": "Human Label" }
  ]
}
```

### Implementation

Modify `generate_codelists.py` to iterate all `str, Enum` subclasses in the models package and emit a codelist JSON file for each. The Pydantic models remain the source of truth; codelists are generated artifacts (like `ojs-strict.json`).

## Non-Goals

- No new modules (artisan/religious/smart/watch/body remain as-is)
- No breaking changes to existing fields
- No field removals
- No changes to required/optional status of existing modules
- `gender` NOT added (covered by ai_commerce.audience_tags)
- `finish_style` NOT added (covered by MetalFinish enum + adding `twisted`)
- `construction_method` NOT added (covered by ArtisanModule.techniques)

## Migration

All additions are backwards-compatible (new optional fields, new enum values). Existing valid v1.0 documents remain valid. SemVer: this is a **minor** version bump (v1.1.0).

## Validation

After implementation:
1. All existing examples must still validate
2. New examples demonstrating new fields must validate
3. `generate_codelists.py` must produce all codelist files
4. `ojs-strict.json` must regenerate with new fields/enums
5. Existing transformer tests must pass unchanged
